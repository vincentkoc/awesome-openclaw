# Contributing

Thanks for helping improve this list.

This repo is curated, not exhaustive. The bar is "useful to real OpenClaw users today," not "contains the keyword OpenClaw."

## Submission Flow

1. Open an issue with the `Add Resource Request` template before opening a PR.
2. Wait for maintainer approval. Unapproved drive-by PRs may be closed.
3. Open a PR with the exact README line you plan to add and link the approved issue.

## Minimum Standards

Submissions should meet all of these:

- Directly relevant to OpenClaw, Moltbot, or Clawdbot usage, development, deployment, skills, plugins, or ecosystem adoption.
- Publicly accessible now: public repo, docs site, product page, or other link people can actually inspect.
- Specific and useful: clear purpose, real functionality, and enough docs or context that someone could decide whether to use it.
- Maintained: recent activity, responsive maintainers, or other evidence the project is not abandoned.
- Supported by at least one meaningful public signal beyond the author's claim.

Acceptable public signals include:

- Meaningful GitHub traction such as stars, forks, watchers, issue/discussion activity, or outside contributors.
- Public usage evidence such as docs, integrations, tutorials, screenshots, case studies, or community write-ups.
- Publicity or ecosystem recognition such as blog posts, community mentions, directory listings, talks, or references from other repos.

Stars are not the only signal, but a project with no visible usage, no publicity, and no traction is usually not a fit.

## Usually Rejected

These are likely to be declined:

- Brand-new repos with no users, no publicity, and no visible ecosystem traction.
- Thin wrappers, copycat forks, or renamed clones without a clear differentiated value.
- Private, invite-only, or inaccessible projects with no public docs or product information.
- SEO pages, placeholder landing pages, or affiliate-style submissions with little real substance.
- Archived, abandoned, or obviously stale projects.
- Duplicate entries or entries that fit a better section without adding new value.

## Higher Bar for Self-Promotion

Self-promo is allowed, hidden self-promo is not.

If you are the founder, maintainer, employee, customer-success rep, or otherwise affiliated with the submission:

- Say so in the issue and PR.
- Expect a higher bar than neutral third-party submissions.
- Bring evidence that the project is already useful beyond your own claims: public users, real docs, integrations, community references, traction, or credible adoption signals.

If the main argument is "I built this," that is not enough.

## What to Include

Please submit a PR with:

- Resource URL.
- Short description: what it does and why it matters.
- Target section.
- Evidence of relevance and public signal.
- Linked approved issue from the `Add Resource Request` template.

## Formatting Rules

- Keep entries in strict A-Z order within each section.
- Keep each entry to a single bullet line with no wrapped sub-bullets.
- Use this format: `- [name](url) - short description.`
- Add status markers only when applicable: `🎖️` for official resources and `💵` for paid services.
- For GitHub repository links, append stars badge: `![GitHub stars](https://img.shields.io/github/stars/owner/repo?style=social)`.
- Keep descriptions factual. Skip hype, marketing copy, and unverifiable claims.

## Commit and PR Conventions

- Use semantic commit prefixes: `feat:`, `fix:`, `docs:`, `chore:`, `refactor:`, `test:`, `ci:`.
- Keep commit messages scoped and specific, for example `docs(readme): add memory plugin integrations`.
- PR titles should also follow semantic format.
- Link the issue in the PR body, for example `Closes #123`.

## Cleanup

PRs are also welcome for:

- Dead links.
- Archived or unmaintained projects that should be removed.
- Better categorization.
- Raising the quality of weak descriptions or misclassified entries.
